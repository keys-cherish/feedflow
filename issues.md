# FeedFlow 已知问题

## 当前已知问题

### 1. RSS 订阅发现失败时的网络问题
- **状态**: 已修复（v0.1.1）
- **描述**: 当 `discover_feed` 进行 HTTP 请求失败时（如 DNS 解析失败、TLS 握手超时、代理配置问题），整个订阅流程会中断
- **修复方案**: 改为 discovery 失败时回退到直接使用原始 URL 进行订阅，而不是直接报错
- **影响范围**: 所有新订阅操作

### 2. 收藏文章视图无专用接口
- **状态**: 已知限制
- **描述**: 后端没有 `GET /api/articles/starred` 端点，前端通过获取全部文章后客户端过滤实现
- **临时方案**: 前端获取 4 倍数量的文章并过滤 `is_starred`
- **建议**: 后续添加 `GET /api/articles/starred` 专用端点

### 3. 文件夹内订阅源的映射关系
- **状态**: 已知限制
- **描述**: 前端刷新页面后，文件夹内的订阅源映射关系丢失（需重新获取）
- **原因**: 没有专用的 `GET /api/folders/{id}/feeds` 端点返回某文件夹下的所有订阅源
- **建议**: 添加文件夹内订阅源列表接口

### 4. 错误信息链路不完整
- **状态**: 已修复（v0.1.1）
- **描述**: API 错误响应只显示最外层 anyhow context，底层错误原因被截断
- **修复方案**: 使用 `{:#}` 格式化 anyhow 错误，显示完整错误链

### 5. favicon.ico 404 错误
- **状态**: 已修复（v0.1.1）
- **描述**: 浏览器自动请求 `/favicon.ico` 返回 404
- **修复方案**: 添加 SVG emoji favicon 处理路由

## 待优化项

### 性能相关
- [ ] 数据库使用 `Mutex<Connection>` 单连接，高并发下可能成为瓶颈
- [ ] 刷新所有订阅源时是串行处理，应改为并发（`tokio::spawn` 或 `FuturesUnordered`）
- [ ] 文章列表未实现虚拟滚动，大量文章时前端可能卡顿

### 功能相关
- [ ] 缺少搜索功能（全文搜索文章）
- [ ] 缺少批量操作（全部已读、批量删除）
- [ ] 缺少文章过期自动清理机制
- [ ] OPML 导入时是串行订阅，大量订阅源导入较慢
- [ ] 缺少订阅源错误状态的 UI 展示

### 安全相关
- [ ] 无用户认证机制，局域网部署需注意安全
- [ ] 文章 HTML 内容已做 ammonia 白名单清洗 + 前端二次清洗，但仍需持续关注 XSS 风险
